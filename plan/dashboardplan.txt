### C. Proposed App Architecture
Design two distinct dashboards with clear separation of concerns:

**Admin Dashboard (for cleanupcrew admin/dispatch team)**
- Sitemap and navigation model
- Lead management: view, create, update lifecycle statuses (new / contacted / quoted / booked / lost), notes, attachments, source tracking, search/filter/bulk actions, export
- Team coordination: view/manage team members, permissions, availability, region assignment
- Dispatch: create assignments from leads/places, assign team members, set schedule windows, view status workflows, track completion
- Reporting: operations (jobs assigned vs completed, cycle time), revenue (by period/place/source), expenses, profit
- Key user journey: lead → place → assignment → completion → invoice → reporting
- Role-based permissions (Admin, Dispatcher, Manager)

**Agent Dashboard (for field staff)**
- Minimal friction, mobile-first design
- View assigned jobs with address, customer details, checklist, and estimated duration
- Status actions: mark accepted/en route/arrived/in progress/completed
- ETA display (manual input, optional GPS integration later)
- Job checklist, before/after photos (optional), notes
- Offline-tolerant UX: queue updates, automatic retry when connectivity restored
- Instant status reflection back to admin dashboard

Include: navigation patterns, role permissions matrix, mobile-first UX principles (fast, minimal taps, clear statuses).

### D. Feature Specifications with Acceptance Criteria

**Leads**
- View/create leads from the existing form and manual creation
- Lead lifecycle statuses: new, contacted, quoted, booked, lost, cancelled
- Lead details, notes, attachments/photos (optional), source tracking (UTM, referrer)
- Search, filters (by status, source, date, notes), bulk actions (export, reassign), export to CSV
- Acceptance criteria: all statuses visible and filterable; search performs within <1s; bulk actions complete without page reload

**Places (Sites/Properties)**
- Full CRUD with address, access instructions, hazards, gate codes (securely encrypted), preferred schedule, recurring options
- Link to leads and jobs/assignments
- Acceptance criteria: one place can link to multiple leads and jobs; all edits logged with timestamp and user; searches include partial address matches

**Team (Staff Directory)**
- CRUD with Canada/US-appropriate fields: name, phone, email, role/designation, emergency contact (optional), availability (days/hours), region/service area, pay type (hourly/salary/contractor), contractor vs employee designation, certifications (optional)
- Roles: Admin, Dispatcher, Manager, Field Agent
- Audit trail: all team member data changes logged with user, timestamp, and old/new values
- Acceptance criteria: non-sensitive fields visible to agents; sensitive data (pay, contractor status) visible only to admins; role changes trigger audit log entry

**Assignments/Dispatch**
- Create jobs from leads/places, assign team members, set schedule window, priority level, estimated duration
- Status workflow: created → assigned → accepted → en route → arrived → in progress → completed → QA → invoiced
- Admin views: list of all assignments with status, map view (optional), filter by agent/place/date/status
- Per-assignment timeline showing status changes, notes, and agent actions
- Acceptance criteria: assignment can be reassigned before agent accepts; status changes are immutable once recorded; admin sees all status updates in <2s

**Agent Field App**
- Receive job assignments with customer address, notes, and estimated duration
- Actions: accept job, mark en route (start ETA timer), mark arrived, start job, mark complete, add notes/photos
- Checklist display and completion tracking
- Before/after photos (optional) with timestamp
- Offline mode: queue actions, sync when connectivity restored
- Realtime reflection: agent's status updates appear in admin dashboard within <5s
- Acceptance criteria: all actions work without network connectivity; photos are compressed to <500KB; status updates sync automatically

**Inventory/Expenses**
- Stock tracking: item name, quantity on hand, reorder threshold, vendor, unit cost, usage per job (optional)
- Purchase orders: create, track delivery, update stock
- Adjustments: log damage, loss, or inventory corrections with reason
- Views: current stock, items below reorder threshold, usage history, purchase order status
- Acceptance criteria: reorder alerts trigger when stock drops below threshold; usage is linked to specific jobs; adjustments are audited

**Invoices & PDFs**
- Generate invoice per property/job with customer name, address, job description, line items, taxes, discounts, deposits
- Invoice statuses: draft, sent, paid, void, partially paid
- CRA-compliant fields: include GST/HST number (if applicable), tax rate, "tax included" vs "added" option, invoice date, due date
- Store PDFs in cloud storage; email/share invoices via link
- Track payment status and log payments
- Acceptance criteria: PDF generates within <3s; invoice includes all CRA-required fields; sent status logged with recipient email and timestamp

**Reporting**
- Operations: jobs assigned vs completed, average completion time, agent productivity (jobs per week), on-time completion rate
- Revenue: total revenue, by period (week/month), by place, by lead source
- Expenses: inventory spend, job-linked costs, total expenses
- Profit: gross margin, net margin (if applicable)
- Exports: CSV, PDF, date range filter
- Acceptance criteria: reports generate within <5s; filters are saved as presets; exports include all visible data

### E. Data Model & Security Architecture

**Supabase Tables & Relationships**
Provide:
- `leads` (id, email, phone, name, address, source, utm_params, status, created_at, updated_at, lead_assigned_to)
- `places` (id, address, access_instructions, hazards, gate_code, recurring_schedule, metadata, created_at, updated_at)
- `lead_place_mapping` (lead_id, place_id)
- `team_members` (id, name, phone, email, role, availability, region, pay_type, contractor_status, certifications, created_at, updated_at)
- `assignments` (id, lead_id, place_id, assigned_to, status, scheduled_window, priority, estimated_duration, created_at, updated_at, completed_at)
- `assignment_status_log` (id, assignment_id, old_status, new_status, changed_by, changed_at) — immutable audit log
- `team_audit_log` (id, team_member_id, field_changed, old_value, new_value, changed_by, changed_at)
- `inventory_items` (id, name, quantity, reorder_threshold, vendor, unit_cost, last_updated_at)
- `inventory_usage` (id, item_id, job_id, quantity_used, used_at)
- `invoices` (id, place_id, assignment_id, status, total_amount, tax_amount, discount_amount, deposit_amount, due_date, created_at, updated_at)
- `invoice_line_items` (id, invoice_id, description, quantity, unit_price, amount)
- `attachments` (id, owner_id, owner_type, file_url, uploaded_at)

**Indexes**: Create indexes on frequently filtered columns (status, assigned_to, created_at, place_id, lead_id, team_member_id).

**Row Level Security (RLS) Policies**
- **Admins**: full read/write access to all tables
- **Dispatchers**: read/write access to leads, places, assignments; read-only access to team and invoicing
- **Managers**: read-only access to leads, assignments, invoicing, reporting; no access to sensitive team fields (pay, contractor status)
- **Field Agents**: read-only access to their assigned jobs; write access only to assignment status changes and notes/photos on their own assignments; no access to leads, team, invoicing, or inventory
- **Public**: no access (all tables require authentication)

Explain each policy clearly with the business logic behind it (least privilege principle).

**Multi-Tenancy & Data Isolation**: If multi-region or multi-account is planned, specify how to organize data. For now, assume single-account multi-role.

**Data Retention & Soft Deletes**
- Soft delete all records (add `deleted_at` column) rather than hard delete
- Retention: 7 years for invoices/financial records (Canada CRA requirement); 2 years for leads; 1 year for completed job history
- Implement automated archival process to move old records to cold storage

**Backup Strategy**: Supabase automates daily backups; ensure backups are tested monthly and retained for 30 days.

### F. Compliance & Privacy Checklist (Canada-First)

**PIPEDA Compliance**
- Consent: Collect explicit consent for email/SMS/phone contact on leads form; store consent status in database
- Collection Limiting: Only collect personal information necessary for job/service delivery
- Safeguards: Encrypt sensitive fields (gate codes, phone, emergency contacts); use RLS to limit access
- Access Rights: Team members can request their personal data; implement data export capability
- Retention: Define retention timeline per data type; delete/archive according to schedule

**CASL Compliance**
- Email/SMS Consent: Leads form must include checkbox for marketing consent (separate from service consent)
- Unsubscribe: Store unsubscribe preferences; honor opt-out requests immediately
- Record Keeping: Log all email/SMS sends with timestamp, recipient, consent status

**Privacy-by-Design Checklist**
- [ ] Sensitive fields (gate codes, pay info, emergency contacts) are encrypted at rest
- [ ] Access to sensitive fields is logged and auditable
- [ ] RLS policies enforce role-based access; admins cannot bypass
- [ ] Audit logs are immutable (events logged to append-only tables)
- [ ] Soft deletes are used; data is never permanently deleted without governance approval
- [ ] Regular access audits: generate monthly report of who accessed what
- [ ] Data minimization: forms only ask for necessary information

**CRA & Tax Compliance (Invoices)**
- Include GST/HST number (if registered) on all invoices
- Show tax rate and total tax collected
- Support "tax included in total" vs "tax added" pricing models
- Keep invoices for 7 years for CRA audit purposes
- Include invoice date, due date, and payment terms

**Important**: You are not a lawyer; present this as product requirements and risk flags for the user to validate with a legal/compliance advisor.

### G. UX/UI System (Mobile-Perfect)

**Design Principles**
- Mobile-first: design for 375px width first, scale up
- Speed: target <2s page load, <500ms interactions
- Clarity: clear status indicators, color-coded priorities, minimal cognitive load
- Accessibility: WCAG 2.1 AA target

**Component Library & Styling**
- Framework: shadcn/ui + Tailwind CSS (or match existing codebase)
- Responsive breakpoints: sm (640px), md (768px), lg (1024px), xl (1280px)
- Color palette: Grayscale + accent color for CTAs and alerts
- Typography: system font stack for speed

**Layout Patterns**
- **Admin Dashboard**: Sidebar navigation (collapsible on mobile) + main content area; top bar for user menu and global actions
- **Agent App**: Bottom navigation (5 tabs max: Home, Jobs, Inventory, Profile, Settings); full-screen job detail view
- **Forms**: Large touch targets (44px minimum), mobile keyboard-aware inputs, progress indicators for multi-step forms
- **Lists**: Cards on mobile, table on desktop; swipe actions for secondary actions (assign, delete, etc.)

**States & Feedback**
- Loading: skeleton screens (not spinners) for better perceived performance
- Empty: contextual empty states with icon, message, and CTA
- Error: clear error messages, retry button, error logging
- Success: toast notifications, automatic dismissal after 3s
- Offline: banner indicating no connectivity; queue indicator for pending actions

**Accessibility Basics**
- Semantic HTML (use `<button>`, `<input>`, `<label>` correctly)
- Color is not the only indicator (use icons + text)
- Form labels are always visible, never placeholder-only
- Focus indicators visible on keyboard navigation
- Alt text on images
